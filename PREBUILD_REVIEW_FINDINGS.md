**TL;DR:** The plan is in genuinely strong shape — every "ports exactly" claim checks out bit-for-bit against the Python code, and the persistence/concurrency/LLM stack claims are almost all confirmed against current sources. The real risk is concentrated in one place: **the media pipeline's geometry/HDR/export assumptions**, several of which are falsified or overstated by Apple's own documentation, and most of which the current phase plan wouldn't test until N12/N16 — long after UI is stacked on the composition architecture. One non-technical landmine: **STTextView is not MIT-licensed**, which D20 asserts and Track 2 depends on. Fixes are cheap now; all of them fit inside a revised N2 gate list plus doc amendments.

## Findings (severity-ranked)

**1. [HIGH] · NATIVE_REWRITE_DECISIONS D20 (round-2 addendum) · STTextView license premise is false — it's GPLv3/commercial, not MIT.**
D20 says "the MIT license permits vendoring the source into the repo if the project ever goes quiet." STTextView's `LICENSE.md` is dual **GPLv3 / paid commercial** ("If you want to use STTextView to develop non open sourced product… the Commercial license is the appropriate license"), and has been since ~6 weeks after the repo's creation. For Track 2 (paid, closed-source direct distribution) that means: buy the commercial license, open-source under GPL, or swap to the NSTextView adapter before shipping. Evidence: [LICENSE.md](https://github.com/krzyzanowskim/STTextView/blob/main/LICENSE.md). **Fix:** correct D20's text now; add a Track 2 line item (license purchase or adapter swap) to N19's scope so it can't be discovered at packaging time. The `TranscriptViewAdapter` seam already caps the swap cost — the decision outcome survives, but the vendoring escape hatch it relied on doesn't exist. (Project health otherwise checks out: v2.3.10 April 2026, active, one maintainer as assumed.)

**2. [HIGH] · PLAN §2.5 rule 5 · Mixed-rotation/geometry footage in one composition track is not displayable correctly with `preferredTransform` alone — FALSIFIED.**
`preferredTransform` is a single track-level value; per-segment transforms require an `AVMutableVideoComposition` with `setTransform:atTime:` layer instructions ([QA1744](https://developer.apple.com/library/archive/qa/qa1744/_index.html)). Portrait iPhone + landscape camera in one track — a real combination in Lillian's footage — breaks rule 5 as written. Cascade the plan doesn't budget: adopting a videoComposition means owning renderSize/frame-duration policy, the HDR color properties (finding 3) ride the same object, and **passthrough export ignores videoComposition transforms** (same QA1744) — so the Lossless tier needs a geometry eligibility check, not just keyframe alignment. **Fix:** decide the geometry policy explicitly (videoComposition from day one vs. normalize rotation at ingest vs. one track per geometry class), add a mixed-rotation render check to N2's gates, and add geometry to export-tier gating in N12.

**3. [HIGH] · PLAN §2.5 rule 8 + D29 · Per-segment HDR→SDR tone-mapping in a bare composition is OVERSTATED, and export defaults the wrong way.**
WWDC20-10009 states mixed-dynamic-range compositions "need additional handling" — the sanctioned mechanism is videoComposition color properties; a bare composition has no documented per-segment tone-mapping guarantee, so HDR↔SDR cuts risk visible shifts in preview. Worse for D29: on export, AVAssetExportSession converts SDR segments **up to HDR** when any HDR is present — the opposite of the default-SDR target — unless color properties are explicitly set. This threatens the WYSIWYG rule in the subtlest way possible (preview tone-maps per display, export flips to HDR). Evidence: [WWDC20-10009](https://developer.apple.com/videos/play/wwdc2020/10009/). **Fix:** explicit color properties on both preview and export paths as a stated requirement; add an HDR↔SDR seam pixel-probe (preview *and* Standard-tier export) to N2's gates. D29's pick (export-time policy, default SDR) survives — it just isn't automatic.

**4. [HIGH] · PLAN §4 N2 vs N12 + D31 · N2's exit gates are preview-only; the export tier structure rests on documented-failure and undocumented behaviors that aren't tested until N12. Add a half-day export spike at N2.**
Specifics: (a) `AVAssetExportPresetPassthrough` on multi-source compositions has a long-standing never-closed radar ([rdar://10421720](https://openradar.appspot.com/10421720)) plus recurring non-deterministic "Operation Stopped" failure reports — there is no Apple documentation promising passthrough works on compositions at all, and whether it authors edit lists or snaps to sync samples in the multi-segment case is publicly unverifiable. (b) The hybrid writer's frame-accuracy mechanism exists (`startSessionAtSourceTime` documents "written but edited out" lead-in semantics) — but only for session start; behavior for **segments 2..N in one pass is undocumented**, the single most spike-worthy unknown in the D31 design. Fallback if it disappoints: per-segment writes stitched with `AVMutableMovie` — currently absent from the plan. (c) MP4-container passthrough requires a non-nil `sourceFormatHint` and supports a restricted subtype set (.mov is the permissive container). (d) Tier 1/1.5 eligibility must gate on **codec/parameter uniformity** — mixed iPhone HEVC + camera H.264 can't pass through into one output track regardless of keyframe alignment. (e) Hand-encoded AAC needs `TrimDurationAtStart` priming attachments or A/V sync drifts ~one frame. (f) Tier 1.5's frame-accuracy is edit-list-based, so it inherits Tier 1's third-party-demuxer caveat — "preview == file" holds in Apple players; the plan should say so rather than implying universality (and the elst failure mode in ignoring demuxers is A/V desync + wrong duration, not just visible lead-in frames). **Fix:** one afternoon at N2 with two iPhone clips: passthrough-on-composition succeed/elst-or-snap check + hybrid-writer segments-2..N edit-out check. Both answers change N12's architecture (one-pass writer vs. AVMutableMovie post-pass), and N12 is already Track 1's biggest phase.

**5. [MED-HIGH] · PLAN §5 N16 + D14 · "Splice via AVAssetWriter" for smart-cut is OVERSTATED — plan the ffmpeg mux path as primary.**
No documented support exists for appending compressed samples with heterogeneous format descriptions to one writer input (the header explicitly requires subsequently-appended buffers to match the format hint), and multi-stsd MP4 support in third-party demuxers is historically poor. Every reference implementation the plan names muxes via ffmpeg/PyAV — [skeskinen/smartcut](https://github.com/skeskinen/smartcut) ("We use PyAV to interface with ffmpeg internals", unmaintained since Feb 2026), [LosslessCut's smart-cut](https://github.com/mifi/lossless-cut/issues/126) (pure ffmpeg and *still* fragile on H.265), avcut (libav). None use AVFoundation. Matched-parameter SPS/PPS splicing is exactly the part mature tools still get wrong. **Fix:** rescope N16 as "smart-cut via ffmpeg mux behind FFmpegLocator (which already exists for .mkv), AVAssetWriter matched-parameter splice as stretch goal." Note this also strengthens the case against dropping the ffmpeg dependency at N19 (D16's "or eliminate by gating .mkv" branch).

**6. [MED-HIGH] · PLAN §2.7 + D21 + N0 kickoff · SPM package targets do NOT inherit Xcode 26's MainActor-default — per-target isolation policy is missing and is a real footgun.**
Per SE-0466, a package target's default isolation is `nonisolated` unless it sets `swiftSettings: [.defaultIsolation(MainActor.self)]` — the Xcode 26 new-target default applies to the app target only, and ~90% of ClipFarm's code lives in ClipFarmKit. If someone "fixes" this by putting MainActor-default on CFDomain/CFMedia, the pure domain functions and background services silently serialize onto the main thread. Evidence: [SE-0466](https://github.com/swiftlang/swift-evolution/blob/main/proposals/0466-control-default-actor-isolation.md). **Fix:** write the policy into the plan/mac-CLAUDE.md: app target = MainActor default; all five ClipFarmKit targets = nonisolated default (Approachable Concurrency upcoming-feature flags opted in per-target); plus one line on SE-0461's `NonisolatedNonsendingByDefault` changing where nonisolated-async runs if enabled asymmetrically across the app/package boundary.

**7. [MED] · PLAN §2.5 rule 4 · "Pre-seek + replaceCurrentItem kills the black-frame blink" is asserted, not established — and the community's actual fix is to avoid `replaceCurrentItem`.**
The blink is a documented, Apple-unanswered issue in precisely this configuration (composition + rebuild-and-swap; [forums 728607](https://developer.apple.com/forums/thread/728607)); the working fix reported there is mutating the current item's composition in place. Pre-seeking an unattached item is legitimate (verify seek completion before swapping). **Fix:** keep rebuild-and-swap as plan A, but make mutate-in-place a *designed* fallback, and gate N2 on a measured swap-blink count (see gate list below).

**8. [MED] · PLAN §2.5/D13/N11 · Boundary time observers are lossier than "fires at exact CMTime boundaries," and <50ms loop restart is optimistic for long-GOP 4K HEVC.**
Documented/reported caveats: fires only when the time is traversed during playback (not while paused, not on seek-past); occasional missed fires; the callback isn't told which boundary fired; and observers must be removed/re-registered after every `replaceCurrentItem` — i.e., after **every edit** in this architecture. Zero-tolerance seeks into long-GOP windows decode from the prior keyframe; practitioner reports describe seek-loop restarts with noticeable delay. **Fix:** re-arm discipline stated in the PlayerEngine contract; belt-and-suspenders end-of-window verification; measure worst-case (4K HEVC, non-keyframe-aligned window) at N2 and either meet 50ms or formally revise the budget. D13's choice (observer over AVPlayerLooper) remains correct.

**9. [MED] · PLAN §3 bit-exact list + spec Phase-10a note · The ported overlap semantics are self-contradictory and make deliberately-overlapping clips un-adjustable.**
Verified in code: `create_clip_from_range` allows overlap (Phase 10a revision, commit aa68b3f), but `adjust_clip_boundaries` rejects any resulting overlap against *every* other clip on the source (boundary.py:295-302, excluding only the moved clip). So the 30s-take + 10s-highlight pair the spec explicitly blesses cannot have either boundary nudged while they still overlap — every adjustment is rejected. The plan would port this bit-exact and N11's trim mode (Library context = boundary correction) would hit it immediately. **Fix:** make a deliberate pre-build decision — e.g., adjust only rejects overlaps that didn't exist before the edit (compare against the pre-edit overlap set), or allow overlap on adjust entirely to match create. Cheap now; a confusing landmine mid-N11 otherwise.

**10. [MED] · PLAN §2.3 schema · The `clip_project_tags` UNIQUE index won't enforce the domain's uniqueness rule for NULL `project_tag_id` — and NULL is the common case.**
SQLite unique indexes treat NULLs as distinct; the Python validator treats `None` as a value and rejects duplicate `(clip, project, None, category)` rows (models.py:249-273, with a named test in test_uniqueness_validator.py). Every bucket-category row (off-topic, fragment, etc.) has NULL `project_tag_id`, so the index is inert exactly where most rows live — the DB would permit what the domain forbids, a silent drift channel if any write path ever bypasses validation. **Fix:** keep validation as the enforcer with a named port test, and/or make the index NULL-proof (e.g., index on `COALESCE(project_tag_id, '')` or a generated column). One line in the plan's bit-exact list.

**11. [MED] · PLAN §2.3/D8 + N1 · `VACUUM INTO` cannot run inside an open transaction — the "snapshot-then-save single locked critical section" invariant needs restating for SQLite mechanics.**
SQLite errors if `VACUUM INTO` runs with a transaction open; in GRDB, `db.vacuum(into:)` (added 5.9.0) must not be called inside a `write { }` block — use a non-transactional/barrier access with the snapshot *preceding* the mutating transaction. Also: an interrupted `VACUUM INTO` can leave a corrupt target file — clean up partials on failure — and it's a full DB copy per destructive op (fine at ClipFarm scale, worth stating). **Fix:** specify the snapshot-then-transaction sequencing in N1's scope so the invariant is defined in terms the storage engine can actually honor.

**12. [MED] · PLAN §2.3 vs D18/D28 · "Per-library" segmentation settings can't live in UserDefaults.**
§2.3 puts settings in UserDefaults, but D18 makes silence threshold/tail policy **per-library**, and D28 makes the library location overridable. UserDefaults is per-app: switch libraries and the settings follow the app, not the library. **Fix:** per-library settings go in the DB (`meta` or a settings table); app-level prefs (provider, key reference, UI state) stay in UserDefaults/Keychain. One-line schema note now, migration later otherwise.

**13. [MED-LOW] · PLAN phases N1–N13 · Several small owned-by-nobody scopes.**
(a) **Snapshot restore** — the spec promises "Settings → Restore snapshot," but no N-phase includes it, and restoring a `.db` (or switching library location, or N13's backup-restore) needs a defined close-connections → swap → reopen path that also **clears the UndoManager stack** and restarts ValueObservations; (b) **backup restore semantics** unspecified (replace vs merge); (c) **`.mkv` remux output location/provenance** unspecified (where the .mp4 lands, which path `sources.path` records); (d) whether an *undo* of a destructive op is itself snapshot-worthy. **Fix:** assign (a)+(b) to N13 (with the reopen path built once, at N1 if convenient), decide (c) in N3's plan, one sentence for (d).

**14. [MED-LOW] · PLAN N3/N4/N5 · FTS5 index maintenance on clip mutations is unstated.**
FTS rows are written at ingest, but N5's split/merge/create/delete/boundary-edit all change or remove `clips.transcript_text` — nothing says the FTS table follows, so search could return deleted clips or stale text (and undo must restore index rows too). **Fix:** external-content FTS5 table with sync triggers (undo-safe by construction) + one test asserting search reflects a split/merge/delete.

**15. [MED-LOW] · PLAN §3 + D8 · Zero planned tests for the undo system that the plan calls a headline invariant.**
"Every mutation undoable, named, coalesced" appears in the invariants, mac/CLAUDE.md, and D8 — and nowhere in the test-parity plan. UndoManager is unit-testable by driving it directly against store methods (mutate → undo → assert domain state and DB round-trip). **Fix:** add "each store mutation lands with a register/undo/redo test" to the §3 strategy; while there, add the resolver to the golden-master list (strategies and segmentation are listed; resolver diffs are equally cheap).

**16. [LOW-MED] · PLAN N14/D26 · Sidecar-fidelity risks for WhisperKit output worth naming now (they shape the N14 acceptance gate).**
(a) The sidecar contract's "words carry a leading space; concatenate raw" is a faster-whisper convention — verify WhisperKit's `WordTiming.word` matches or the transcript view renders "wordword"; (b) turbo's pruned 4-layer decoder gives fewer DTW alignment heads — word-timing *quality* (not support) is the soft spot; N14's gate should golden-master word timings against existing `transcribe.py` sidecars on the dogfood folder, not just WER; (c) if the SpeechTranscriber fallback is ever wired, its timing is run-level AttributedString attributes, not a per-word array — the sidecar writer needs a flattening adapter. (WordTiming shape `{word, tokens, start, end, probability}`, macOS 14+, and ≥10× realtime all otherwise confirmed.)

**17. [LOW] · PLAN §2.7/D16 · swift-subprocess is pre-1.0 with fresh breaking churn.**
0.5 (May 2026) restructured the core API; 1.0.0-beta.1 landed 2026-07-02. Pin the exact version; expect one migration at 1.0. Also: collected-output modes **throw** when the child exceeds the byte limit — ffmpeg's chatty stderr needs a generous limit or the streaming API.

**18. [LOW] · PLAN N3 · Waveform generation for every source at ingest must be stated as background work.**
A full-folder ingest (18× 30-min sources) with synchronous audio decode would turn seconds into minutes; the WaveformService exists — N3 should say waveforms are generated async post-ingest, with N11 degrading gracefully if a strip isn't ready.

**19. [LOW] · Doc hygiene (batch).**
(a) PLAN §9.11 still says "Track 2 (N14–N18)" — stale after the round-2 renumbering to N14–N19; (b) DECISIONS D14 references "round 3" while both doc headers describe two rounds; (c) plan/spec wording "refuses a **higher** schema_version" — code rejects any `schema_version != 1` including lower/missing (stricter; align wording); (d) §2.3's `projects.script_lines` column vs. amendment #8's `script` naming — fine at rest, but note the backup exporter must emit the documented `script: {lines: []}` shape; (e) D6's "documented undo-related crashes" (plural) on macOS 26 is real but rests on one Apple-acknowledged report — cite FB22539495 + the UndoManager-leak thread ([822241](https://developer.apple.com/forums/thread/822241), [756757](https://developer.apple.com/forums/thread/756757)) so the premise is anchored; (f) unverified caution on D24: "stable signing identity makes TCC grants persist" holds only with a real Apple Development certificate — ad-hoc "Sign to Run Locally" re-signs per build and re-triggers TCC prompts; worth confirming the signing setup at N0.

## Consolidated N2 gate additions (the direct answer to "are N2's gates the right gates?")

The existing five gates are right but insufficient — they'd pass while findings 2, 3, 4, 7 stay latent until N10–N12. Add:

1. **Seam-drop instrumentation:** 20+ deliberately non-keyframe-aligned cuts across ≥3 files (ProRes + H.264 + HEVC, 4K); capture frame-delivery timestamps; gate p95 inter-frame gap at seams ≤ 1 frame duration. (Apple's own documented mitigation if this fails is *two alternating video tracks* — A/B that before anything custom.)
2. **Swap-blink count:** 100 edit→rebuild→pre-seek→swap cycles under screen capture, A/B'd against mutate-in-place; gate: zero visible blinks on the chosen strategy.
3. **Mixed-rotation render check:** portrait iPhone + landscape camera in one composition — confirms the videoComposition requirement and what passthrough does with it.
4. **HDR↔SDR seam probe:** alternating HLG/SDR segments, with and without videoComposition color properties, pixel-probed in preview *and* a Standard-tier export; gate: no visible shift and preview == export.
5. **Worst-case trim-loop restart:** boundary-fire → first rendered frame of window start on long-GOP 4K HEVC; meet 50ms or revise the budget.
6. **End-to-end edit latency:** edit → new item `readyToPlay` → first frame, 50-clip composition, warm cache — this, not edit-list-construction milliseconds, is the number that matters.
7. **Export mini-spike (half a day):** (a) passthrough export of a two-file H.264 composition with non-keyframe cuts — succeeds at all? edit lists or snapping? (b) hybrid-writer sequential-session test — are lead-in frames edited out for segments 2..N? (c) quick elst A/B of one output in QuickTime/VLC/Chrome/Resolve-import.
